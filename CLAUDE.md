# にゃんポーカー プロジェクト

## 開発環境

- Python: uv で管理
- パッケージ追加: `uv add <package>`
- スクリプト実行: `uv run <script.py>`

## プロジェクト構成

- `docs/`: 仕様書
- `images/`: 猫の画像とmap.yml
- `mock/`: UX/UIモックファイル
- `tmp/`: 一時ファイル・分析用スクリプト

## 実装ルール

### UX/UIの一貫性確認

アプリ実装時は、必ず `mock/index.html` を参照してUXが壊れていないかを確認すること。

- 画面遷移、アニメーション、インタラクションはモックに準拠する
- デザイン（色、サイズ、レイアウト）はモックを正とする
- モックと異なる実装をする場合は、先にモックを更新してから実装する

## GitHub

### ブランチ作成ルール

#### 命名規則

```
<type>/<issue番号>-<簡潔な説明>
```

#### typeの種類

- `feature/`: 新機能追加
- `fix/`: バグ修正
- `docs/`: ドキュメント更新
- `refactor/`: リファクタリング
- `test/`: テスト追加・修正
- `chore/`: その他（設定変更、依存関係更新など）

#### 例

- `feature/12-add-login-screen`
- `fix/45-card-display-bug`
- `docs/8-update-readme`
- `refactor/23-extract-game-logic`

#### ルール

- 説明部分はケバブケース（小文字、ハイフン区切り）を使用する
- Issue番号がない場合は省略可: `feature/add-new-animation`
- 短く具体的に記述する（長くても30文字程度）
- 日本語は使用しない（英語のみ）

### Issue登録ルール

Issueを登録する際は以下のルールに従うこと。

#### タイトル

- 簡潔かつ具体的に記述する
- プレフィックスを付ける: `[Bug]`, `[Feature]`, `[Improvement]`, `[Docs]` など

#### 本文テンプレート

**Bugの場合:**
```markdown
## 概要
（バグの簡潔な説明）

## 再現手順
1.
2.
3.

## 期待する動作
（本来どうあるべきか）

## 実際の動作
（現在何が起きているか）

## 環境
- OS:
- Python:
- その他関連情報:
```

**Featureの場合:**
```markdown
## 概要
（機能の簡潔な説明）

## 背景・目的
（なぜこの機能が必要か）

## 要件
- [ ] 要件1
- [ ] 要件2

## 参考
（関連するドキュメント、モック、画像など）
```

#### ラベル

適切なラベルを付与する:
- `bug`: バグ報告
- `enhancement`: 機能追加・改善
- `documentation`: ドキュメント関連
- `question`: 質問・相談
- `priority:high/medium/low`: 優先度

### プルリクエスト作成ルール

プルリクエストを作成する際は以下のルールに従うこと。

#### タイトル

- 変更内容を簡潔に記述する
- 関連するIssue番号がある場合は末尾に記載: `Add card data generation (#3)`

#### 本文テンプレート

```markdown
## Summary

（変更内容の概要を1-3行で記述）

## Changes

- 変更点1
- 変更点2
- 変更点3

## Code Review Results

### 指摘事項と修正内容

| # | 指摘事項 | 対応内容 | 対応状況 |
|---|---------|---------|---------|
| 1 | （レビューで指摘された内容） | （どのように修正したか） | ✅ |
| 2 | （レビューで指摘された内容） | （どのように修正したか） | ✅ |

※ コードレビューで指摘がなかった場合は「指摘事項なし」と記載

## Test Results

- テスト実行結果: ✅ All tests passed (XX/XX)
- カバレッジ: XX%

## Related Issues

- Closes #XX （または Fixes #XX, Resolves #XX）

## Checklist

- [ ] コードレビュー実施済み
- [ ] テスト追加・更新済み
- [ ] mock/index.html との整合性確認済み（UI変更がある場合）
```

#### 注意事項

- コードレビュー（code-reviewer-ja）を実施してからPRを作成すること
- レビュー指摘事項は必ず記録し、対応内容を明記すること
- テスト結果（成功数/総数、カバレッジ）を必ず記載すること
- UI変更がある場合はスクリーンショットを添付すること
